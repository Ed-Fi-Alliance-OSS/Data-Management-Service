# DMS-987 Code Review

### Correctness risks / gaps
1. **POST upsert skips referential-identity maintenance for existing descriptors (Medium)** — **What's wrong:** when a descriptor POST resolves to an existing document, the write path updates only `dms.Descriptor` and `dms.Document`, leaving `dms.ReferentialIdentity` untouched even though the story requires inserting/updating the descriptor referential-identity row for POSTs. **Evidence:** `reference\design\backend-redesign\epics\07-relational-write-path\06-descriptor-writes.md:21-24,40-42` (POST must insert/update referential identity) vs `src\dms\backend\EdFi.DataManagementService.Backend\DescriptorWriteHandler.cs:337-366,444-486` (POST-upsert update SQL only touches `dms.Descriptor` and `dms.Document`). **Impact:** if the referential-identity row is missing or stale (e.g., seeded/migrated data or partial backfills), POST upserts won’t repair it, so reference resolution remains broken and upserts can devolve into write conflicts instead of updates. **Recommendation:** when the POST target context is `ExistingDocument`, upsert the `dms.ReferentialIdentity` row for `(DocumentId, ResourceKeyId)` using the normalized referential id (Postgres `INSERT ... ON CONFLICT ... DO UPDATE`, SQL Server `MERGE`), or explicitly validate and repair the row before returning success.

### Design/spec drift
1. **Descriptor `Uri` is stored and compared without Core normalization (High)** — **What's wrong:** the specs require normalized descriptor URIs (Core lowercases descriptor identities and reference URIs), but the write path derives `Uri` from raw `namespace` + `codeValue` and uses that raw value for immutability checks; tests explicitly assert “preserve original case.” **Evidence:** `reference\design\backend-redesign\epics\07-relational-write-path\06-descriptor-writes.md:10-14,40-42` and `reference\design\backend-redesign\epics\02-ddl-emission\05-descriptor-ddl.md:19-24` (normalized URI requirement) vs Core normalization in `src\dms\core\EdFi.DataManagementService.Core\Model\DescriptorDocument.cs:26-45` and `src\dms\core\EdFi.DataManagementService.Core\Extraction\DescriptorExtractor.cs:62-70`, plus current implementation `src\dms\backend\EdFi.DataManagementService.Backend\DescriptorWriteBodyExtractor.cs:56-76` and `src\dms\backend\EdFi.DataManagementService.Backend\DescriptorWriteHandler.cs:164-168`, with tests asserting original case in `src\dms\backend\EdFi.DataManagementService.Backend.Tests.Unit\DescriptorWriteBodyExtractorTests.cs:99-115`. **Impact:** case-only changes in `namespace`/`codeValue` are incorrectly rejected as identity changes, and `dms.Descriptor.Uri` diverges from the normalized identity used by referential-id resolution, which can break downstream queries/diagnostics and violates the documented normalization contract. **Recommendation:** normalize descriptor URIs using the same lowercasing rules as Core before persistence and identity comparison; update the extractor/handler and tests to expect the normalized `Uri` value.
