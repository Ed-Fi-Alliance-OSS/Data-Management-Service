# DMS-978: Configuration + Fail-Fast Behaviors for Schema/Pack Selection

## Overview

**Jira**: [DMS-978](https://edfi.atlassian.net/browse/DMS-978)
**Parent**: DMS-974 — Runtime Schema Validation & Mapping Set Selection (06)
**Branch**: `DMS-978`
**Predecessor**: DMS-977 (completed — mapping set selection infrastructure)

This story adds strongly-typed configuration, actionable diagnostics, structured logging, and multi-instance-safe failure modes to the mapping set selection pipeline built in DMS-977.

---

## Acceptance Criteria (from design doc)

1. Configuration supports (at minimum): mapping pack enable/required toggles, pack root path, allow runtime compile fallback, cache mode/capacity.
2. Failures include actionable diagnostics: DB `EffectiveSchemaHash`, expected mapping version/dialect, and whether a pack was required/missing/invalid.
3. A schema mismatch for one database does not prevent DMS from serving other databases (multi-instance-safe failure mode).

---

## Current State

### What DMS-977 delivered
- `MappingSetProviderOptions` — simple POCO with 4 properties (`PacksEnabled`, `PacksRequired`, `PackRootPath`, `AllowRuntimeCompileFallback`)
- `MappingSetProvider` — coordinates pack loading, runtime compilation fallback, and caching
- `MappingSetCache` — concurrency-safe in-process cache (ConcurrentDictionary + TaskCompletionSource), **unbounded**
- `ResolveMappingSetMiddleware` — request-time mapping set resolution with 503 error handling
- `MappingSetUnavailableException` — single exception type, message-only (no structured data)
- DI registration: `services.Configure<MappingSetProviderOptions>(_ => { })` (empty defaults, no appsettings binding)

### What DMS-978 needs to deliver
- Bind `MappingSetProviderOptions` to an `appsettings.json` section
- Add cache configuration properties
- Options validation at startup
- Standardize error/exception types with actionable diagnostics
- Add structured logging throughout the selection/validation flow
- Unit tests for configuration combinations and error behaviors

## Design Decisions

### DD-1: Configuration Section Name

**What the design doc says**: `aot-compilation.md` section 7.2 proposes a `"MappingPacks"` section:
```json
{
  "MappingPacks": {
    "Enabled": true,
    "Required": false,
    "RootPath": "/etc/dms/mapping-packs",
    "CacheMode": "InMemory",
    "AllowRuntimeCompileFallback": true
  }
}
```

**Current codebase pattern**: All appsettings section names use PascalCase and are descriptive nouns/noun-phrases: `AppSettings`, `CircuitBreaker`, `DeadlockRetry`, `CacheSettings`, `ConfigurationServiceSettings`, `DatabaseOptions`, `JwtAuthentication`, `RateLimit`.

**Decision**: Use `"MappingPacks"` as prescribed by the design doc. It follows the existing PascalCase convention and matches the conceptual domain.

---

### DD-2: Property Renaming (or Not)

**What the design doc says**: The AOT doc uses short names (`Enabled`, `Required`, `RootPath`) without a `Pack` prefix — because the section itself is already named `MappingPacks`.

**Current codebase pattern**: The existing `MappingSetProviderOptions` uses `PacksEnabled`, `PacksRequired`, `PackRootPath` — which made sense when there was no configuration section (the prefixes disambiguated from the cache/compiler settings).

**Options**:
- **(A) Rename**: `PacksEnabled` -> `Enabled`, `PacksRequired` -> `Required`, `PackRootPath` -> `RootPath`. Aligns with design doc. Clean JSON (`MappingPacks:Enabled`). But requires updating all consumers (tests, provider code).
- **(B) Keep current names**: Avoid churn. JSON becomes `MappingPacks:PacksEnabled` — redundant but not harmful.

**Suggestion**: **(A) Rename**. The property names are redundant once nested under `MappingPacks`, and consumers are limited (~3 production files, ~1 test file). The design doc explicitly uses the shorter names, and the churn is small.

---

### DD-3: Cache Configuration — What to Expose

**What the design doc says**: Acceptance criteria list "cache mode/capacity". The AOT doc proposes `CacheMode: "InMemory"` and doesn't elaborate on capacity.

**Current codebase patterns for caching**:
- **No caches in the codebase have explicit capacity limits.** All ConcurrentDictionary-based caches (CompiledSchemaCache, MappingSetCache, DatabaseFingerprintProvider, ResourceKeyValidationCacheProvider, NpgsqlDataSourceCache) are unbounded.
- **HybridCache-based caches** (CachedApplicationContextProvider, CachedClaimSetProvider, CachedProfileService) use TTL-based eviction but no explicit capacity.
- **CacheSettings** only exposes TTL values (seconds), not capacity.

**Context for MappingSetCache**: Each entry is keyed by `(EffectiveSchemaHash, Dialect, RelationalMappingVersion)`. In practice, the number of unique keys is bounded by `(number of distinct DB schemas) × (number of dialects)` — typically 1-5 entries in production. Even aggressive multi-instance deployments are unlikely to exceed ~50. A mapping set is large (holds all compiled SQL plans), so each entry is non-trivial in memory.

**Options**:
- **(A) Add CacheCapacity**: A configurable max entry count. Reject new entries at capacity.
- **(B) Defer capacity to a future story**: Since no existing cache has capacity limits and the practical key space is small, this might be premature.
- **(C) Add CacheMode only**: Accept the design doc's `CacheMode: "InMemory"` as a future-proofing enum but don't implement alternatives yet. Defer capacity.

**Suggestion**: **(C) Add CacheMode only, defer capacity**. Reasoning:
- No existing cache in the codebase has capacity limits — adding one here would be inconsistent.
- The key space is naturally bounded and small.
- `CacheMode` is a useful extensibility point (future: `Disabled`, `InMemory`, `Distributed`).
- Adding capacity now would require deciding reject-vs-evict semantics without real-world data.
- If we want to add capacity later, it's a backwards-compatible addition.

**`CacheMode` enum**:
```csharp
public enum MappingSetCacheMode
{
    InMemory // Default and only supported mode for now
}
```

### DD-4: Options Validation Pattern

**Current codebase pattern**: The project has **7 `IValidateOptions<T>` implementations** across Frontend and Config Service. The standard pattern is:

```csharp
// Registration:
services.Configure<T>(configuration.GetSection("SectionName"))
        .AddSingleton<IValidateOptions<T>, TValidator>();

// Trigger at startup (in Program.cs/WebApplicationBuilderExtensions):
_ = app.Services.GetRequiredService<IOptions<T>>().Value;

// Invalid config -> ReportInvalidConfigurationMiddleware short-circuits all requests
```

Validators use `ValidateOptionsResult.Fail(message)` for hard failures.

**What to validate**:
- `Required == true && Enabled == false` -> Fail (contradictory)
- `CacheMode` is a defined enum value -> Fail on undefined
- `AllowRuntimeCompileFallback == false && Enabled == false` -> Warning only (fallback setting is irrelevant when packs are disabled; not an error, just noise)

**Note on `RootPath` validation**: The design doc says `RootPath` is only used when `Enabled=true`. But the actual file-based pack store (`IMappingPackStore`) is not yet implemented (DMS-968 — `NoOpMappingPackStore` is the current stub). Validating `RootPath` existence now would be premature — it would fail in every current deployment. Leave `RootPath` validation to the file-based store implementation.

**Suggestion**: Follow the established `IValidateOptions<T>` pattern. Register in the same DI chain where options are bound. Only validate internally-contradictory settings, not downstream dependencies.

### DD-5: DI Binding Location

**Current state**: `MappingSetProviderOptions` is registered in `PostgresqlServiceExtensions.cs` with an empty lambda:
```csharp
services.Configure<MappingSetProviderOptions>(_ => { });
```

**Current codebase pattern**: Options that are cross-cutting or used by Core middleware are bound in `WebApplicationBuilderExtensions.cs` (Frontend) using `configuration.GetSection(...)`. Backend-specific options are bound in backend extension methods.

**The issue**: `MappingSetProviderOptions` is consumed by:
- `MappingSetProvider` (Backend.Plans) — backend layer
- `ResolveMappingSetMiddleware` doesn't consume it directly but its behavior is shaped by it

The options class lives in `Backend.External`, which is the contracts layer between Core and Backend. It's currently registered in the PostgreSQL backend, but it's dialect-neutral — the same options apply for MSSQL too.

**Impact analysis**: `AddPostgresqlDatastore()` is called in 3 places:

| # | Location | Has `IConfiguration`? |
|---|----------|-----------------------|
| 1 | `WebApplicationBuilderExtensions.cs:186` | Yes — `webAppBuilder.Configuration` |
| 2 | `PostgresqlRuntimeMappingInitializationTests.cs:204` | No — needs 1 line to build empty config |
| 3 | `PostgresqlRuntimeMappingInitializationTests.cs:1317` | Yes — `configuration` already on line 1301 |

No E2E impact (E2E uses `WebApplicationFactory` which goes through the full `AddServices()` path). No Config Service impact (different class in a different namespace).

**Options**:
- **(A) Move to WebApplicationBuilderExtensions.cs**: Where other cross-cutting options are bound. **Breaks** test setups that call `AddPostgresqlDatastore()` directly — they'd lose `IOptions<MappingSetProviderOptions>` entirely.
- **(B) Keep in PostgresqlServiceExtensions.cs, add overload**: New overload `AddPostgresqlDatastore(IServiceCollection, IConfiguration)` for production; keep parameterless for tests. Adds complexity: two public methods + a private shared method.
- **(C) Replace parameterless with IConfiguration parameter**: Change signature to require `IConfiguration`. 3 one-line call-site fixes (trivial).

**Decision**: **(C) Replace with `IConfiguration` parameter**. Reasoning:
- One method with a clear contract is cleaner than an overload + private `Core` extraction.
- The mapping set compilation pipeline (MappingSetCompiler, IMappingPackStore, IRuntimeMappingSetCompiler, IMappingSetProvider) is all registered in `AddPostgresqlDatastore()`. The options belong alongside the services that consume them.
- Only 3 call sites need updating, and the test fix is trivial (test at line 1317 already has `configuration`; test at line 204 needs one line to build an empty `IConfiguration`).
- Eliminates the risk of someone accidentally calling a parameterless overload in production and getting silent defaults instead of real config.
- The `IValidateOptions` validator and its startup trigger are registered separately in `WebApplicationBuilderExtensions.cs`, since that's where all validators live and where `ReportInvalidConfiguration()` runs.

**Concrete change**:
```csharp
// PostgresqlServiceExtensions.cs — single method, requires IConfiguration
public static IServiceCollection AddPostgresqlDatastore(
    this IServiceCollection services, IConfiguration configuration)
{
    services.Configure<MappingSetProviderOptions>(configuration.GetSection("MappingPacks"));
    services.AddSingleton<MappingSetCompiler>();
    services.AddSingleton<IMappingPackStore, NoOpMappingPackStore>();
    // ... rest of existing registrations unchanged
}
```

```csharp
// WebApplicationBuilderExtensions.cs — production caller + validator
webAppBuilder.Services.AddPostgresqlDatastore(webAppBuilder.Configuration);

services.AddSingleton<IValidateOptions<MappingSetProviderOptions>,
    MappingSetProviderOptionsValidator>();

// In ReportInvalidConfiguration():
_ = app.Services.GetRequiredService<IOptions<MappingSetProviderOptions>>().Value;
```

```csharp
// PostgresqlRuntimeMappingInitializationTests.cs line 204 — add config
var configuration = new ConfigurationBuilder().AddInMemoryCollection([]).Build();
services.AddPostgresqlDatastore(configuration);

// PostgresqlRuntimeMappingInitializationTests.cs line 1317 — pass existing config
.AddPostgresqlDatastore(configuration)
```

### DD-6: Error/Exception Strategy — Structured Failure Data

**Current state**: `MappingSetUnavailableException` carries only a `string message` and optional `Exception innerException`. The middleware constructs a generic 503 response that says "check server logs for details."

**What the design doc says**: "Failures include actionable diagnostics: DB `EffectiveSchemaHash`, expected mapping version/dialect, and whether a pack was required/missing/invalid."

**Current codebase patterns for exceptions with structured data**:

The closest precedent is `DatabaseFingerprintValidationException`:
```csharp
public sealed class DatabaseFingerprintValidationException : InvalidOperationException
{
    public IReadOnlyList<string> ValidationIssues { get; }
    // Multiple constructors, validates inputs
}
```
The middleware extracts `ex.ValidationIssues` and passes them as the `errors` array in the `FailureResponse`.

**Options**:
- **(A) New record + enum on the exception**: Add `MappingSetSelectionFailure` record and `MappingSetFailureReason` enum as a property on the exception. Rich type-safe data.
- **(B) Follow DatabaseFingerprintValidationException pattern**: Add an `IReadOnlyList<string>` of diagnostic details (string-based). Simpler, matches existing precedent exactly.
- **(C) Just improve the message strings**: Don't add structured data — make the existing `message` strings more actionable. Least churn.

**Suggestion**: **(B) Follow the `DatabaseFingerprintValidationException` pattern**. Reasoning:
- It's the established pattern for exceptions that carry diagnostic details to the middleware layer.
- The middleware already knows how to handle `string[] errors` via `FailureResponse.ForMappingSetUnavailable(title, detail, errors, traceId)`.
- An enum adds a type that the middleware must understand, but the middleware doesn't *act* differently per reason — it always returns 503 with the errors array.
- The throw sites already have rich, actionable messages. Adding a `string[] diagnostics` property lets the middleware surface them without parsing messages.

**Concrete change**:
```csharp
public sealed class MappingSetUnavailableException : Exception
{
    public IReadOnlyList<string> Diagnostics { get; }

    public MappingSetUnavailableException(string message, IReadOnlyList<string> diagnostics)
        : base(message) { Diagnostics = diagnostics; }

    public MappingSetUnavailableException(string message, IReadOnlyList<string> diagnostics, Exception innerException)
        : base(message, innerException) { Diagnostics = diagnostics; }

    // Keep existing constructors for backwards compat (Diagnostics defaults to [message])
    public MappingSetUnavailableException(string message)
        : base(message) { Diagnostics = [message]; }

    public MappingSetUnavailableException(string message, Exception innerException)
        : base(message, innerException) { Diagnostics = [message]; }
}
```

At throw sites, construct actionable diagnostic strings:
```csharp
throw new MappingSetUnavailableException(
    message: $"Mapping pack is required but not found for {FormatKeyForMessage(key)}.",
    diagnostics: [
        $"EffectiveSchemaHash: {key.EffectiveSchemaHash}",
        $"Dialect: {key.Dialect}",
        $"RelationalMappingVersion: {key.RelationalMappingVersion}",
        "Pack status: required but not found",
        "Suggested action: Provide a matching .mpack file or set Required=false."
    ]
);
```

In the middleware, pass `ex.Diagnostics` as the `errors` array:
```csharp
Body: FailureResponse.ForMappingSetUnavailable(
    MappingSetUnavailableTitle,
    ex.Message,
    [.. ex.Diagnostics],
    requestInfo.FrontendRequest.TraceId
)
```

### DD-7: Structured Logging Strategy

**Current codebase patterns**:
- Named parameters use PascalCase: `{EffectiveSchemaHash}`, `{Dialect}`, `{TraceId}`
- Every log includes `TraceId` for request correlation
- `LogSanitizer.SanitizeForLog()` (Backend.External) for backend components
- `LoggingSanitizer.SanitizeForLogging()` (Core) for middleware/core components — wraps `LogSanitizer`
- `LogError(ex, template, ...)` pattern includes exception as first parameter
- `MappingSetProvider` already has structured logs at Information level (pack loaded, fallback, compiling)

**What to add**:
- The existing logs in `MappingSetProvider` are already decent. The gap is:
  1. No `Debug` log for cache hits (to diagnose "is compilation happening repeatedly?")
  2. No `Warning` log before throwing on fail-fast paths (thrown exceptions go to `Error` in the middleware, but a `Warning` at the decision point helps trace the flow)
  3. Runtime compilation success isn't logged (only the start is logged)

**Suggestion**: Add targeted logs at the gaps, don't duplicate what's already there:
- `Debug`: "Mapping set cache hit for {EffectiveSchemaHash}, {Dialect}" — in `MappingSetCache.GetOrCreateAsync` when the cache entry already exists
- `Warning`: Before each `throw` in fail-fast paths (pack required but missing, fallback disabled)
- `Information`: After successful runtime compilation ("Runtime mapping set compiled successfully for ...")
- Keep using `LogSanitizer.SanitizeForLog()` in Backend components (consistent with existing usage)

### DD-8: Multi-Instance Safety

**What the design doc says**: "A schema mismatch for one database does not prevent DMS from serving other databases."

**Current state**: Already supported by design:
- `MappingSetCache` keys are per `(EffectiveSchemaHash, Dialect, RelationalMappingVersion)` — independent entries
- Failure evicts only the failed cache entry (line 89-96 in `MappingSetCache.cs`):
  ```csharp
  catch (Exception ex)
  {
      cache.TryRemove(new KeyValuePair<MappingSetKey, CacheEntry>(key, this));
      _completionSource.TrySetException(ex);
  }
  ```
- This matches the pattern in `DatabaseFingerprintProvider` and `ResourceKeyValidationCacheProvider`

**Suggestion**: No production code changes needed — the cache already isolates failures per key. Add an explicit unit test that verifies: "given compilation fails for key A, a subsequent call for key B still succeeds." This documents the contract and prevents regressions.

---

### DD-9: `FailureResponse` Changes

**Current state**: `FailureResponse.ForMappingSetUnavailable(title, detail, errors, traceId)` exists and produces a standard problem+json response with type `urn:ed-fi:api:mapping-set-unavailable`.

**What the design doc wants**: The 503 response should include EffectiveSchemaHash, dialect, mapping version, and pack status.

**Current pattern**: The middleware currently passes hardcoded strings to `errors[]`:
```csharp
errors: ["The compiled mapping set for this database is unavailable. Check server logs for details."]
```

**Suggestion**: No changes to `FailureResponse` itself. The `errors` array already supports multiple strings. The middleware just needs to pass `ex.Diagnostics` instead of a hardcoded message. The `detail` field gets the exception message. This keeps the response structure consistent with `ForDatabaseFingerprintValidationError` which uses the same `errors` array pattern.
