# DMS-953: Emit `ddl.manifest.json` (Deterministic DDL Summary)

## Jira

- **Ticket:** [DMS-953](https://edfi.atlassian.net/browse/DMS-953)

- **Referenced By:** `reference/design/backend-redesign/design-docs/ddl-generator-testing.md`, `reference/design/backend-redesign/design-docs/ddl-generation.md`

## Summary

Emit a deterministic `ddl.manifest.json` artifact alongside `{dialect}.sql` during DDL emission. The manifest provides a stable per-dialect DDL summary (normalized SQL hashes and statement counts) for diagnostics and fast diffs without relying solely on large SQL snapshots.

## Acceptance Criteria (from Jira)

1. When enabled (fixture or CLI option), `ddl.manifest.json` is emitted alongside `{dialect}.sql`.
2. The manifest is deterministic (byte-for-byte stable for the same inputs) and uses `\n` line endings only.
3. For each dialect emitted, the manifest includes at minimum:
   - `dialect` id (`pgsql`/`mssql`)
   - SHA-256 (or equivalent) of the **normalized** SQL text
   - `statement_count` (post-normalization, using a deterministic splitter)
   - Any additional stable metadata needed for troubleshooting (optional)
4. Snapshot/golden tests validate `ddl.manifest.json` output for at least one small fixture.

## Manifest Schema (Normative)

Per `ddl-generator-testing.md`:

```json
{
  "effective_schema_hash": "<effective-schema-hash>",
  "relational_mapping_version": "<relational-mapping-version>",
  "ddl": [
    {
      "dialect": "mssql",
      "normalized_sql_sha256": "<64-char lowercase hex SHA-256>",
      "statement_count": 42
    },
    {
      "dialect": "pgsql",
      "normalized_sql_sha256": "<64-char lowercase hex SHA-256>",
      "statement_count": 38
    }
  ]
}
```

- `ddl[]` is ordered by dialect name (ordinal ascending: `mssql` before `pgsql`).
- `normalized_sql_sha256` is a lowercase hex string (64 chars) of the SHA-256 of the normalized SQL text bytes (UTF-8, no BOM).
- `statement_count` is the number of statements after deterministic splitting.
- JSON is indented with `\n` line endings and a trailing newline.

## Implementation Plan

### Phase 1: Define Normalization Rules and Manifest Schema

**Normalization rules for SQL hashing:**

The SQL text emitted by the DDL pipeline (`CoreDdlEmitter` + `RelationalModelDdlEmitter` + `SeedDmlEmitter`) already goes through `SqlWriter.Canonicalize()` which produces:
- Unix LF only (`\n`)
- No trailing whitespace on any line
- 4-space indentation

For `ddl.manifest.json`, the normalized SQL text is this canonical output **as-is**. No further transformation is needed because the emitters already guarantee deterministic, canonicalized output.

**Statement counting rules (deterministic splitter):**

Statements are counted by splitting the normalized SQL text:
1. Split on `;` at end of a statement (ignoring `;` inside string literals, comments, or trigger/function bodies)
2. For SQL Server, also recognize `GO` batch separators as statement boundaries
3. A simpler approach: since our emitters produce well-structured DDL, count the number of top-level DDL/DML statements by counting lines matching the pattern of statement terminators (`;` at end of non-empty trimmed line, plus `GO` for MSSQL)
4. Empty segments (whitespace-only) are not counted

**Validated counting algorithms** (T1 finding — naive `;\n` counting is incorrect because trigger/function bodies contain internal `;`):

**MSSQL:**
1. Split the SQL text by standalone `GO\n` lines into segments
2. First segment (plain DDL/DML before any GO): count `;\n` occurrences
3. Each subsequent non-empty segment (trigger batch): count as 1 statement
4. Total = first-segment count + number of non-empty GO-delimited batches

**PostgreSQL:**
1. Track `$$` delimiter state using line-boundary heuristics:
   - Line ends with `$$` (e.g., `DO $$`, `AS $$`) → toggle into dollar-quoted block
   - Line starts with `$$` (e.g., `$$ LANGUAGE plpgsql;`) → toggle out of dollar-quoted block
   - `END $$;` → toggle out of dollar-quoted block
   - `$$` mid-line inside string literals (e.g., `'$$.schoolId='`) → ignored
2. Count `;\n` only when outside `$$` blocks

**Verified against all 6 golden fixtures** — see `task-953.json` for exact counts per fixture

### Phase 2: Implement `DdlManifestEmitter`

**File:** `src/dms/backend/EdFi.DataManagementService.Backend.Ddl/DdlManifestEmitter.cs`

**Responsibilities:**
- Accept a list of `(dialect, combinedSqlText)` tuples
- Accept the `EffectiveSchemaInfo` (for `effective_schema_hash` and `relational_mapping_version`)
- For each dialect entry:
  - Compute SHA-256 of the normalized SQL text (UTF-8, no BOM) → lowercase hex (64 chars)
  - Count statements using the deterministic splitter
- Emit deterministic JSON using `System.Text.Json.Utf8JsonWriter` (same pattern as `EffectiveSchemaManifestEmitter`)
- Ensure `ddl[]` array is ordered by dialect name (ordinal)
- JSON indented with `\n` line endings, trailing newline

**Key design decisions:**
- Follow the same `Utf8JsonWriter` pattern used by `EffectiveSchemaManifestEmitter` and `DerivedModelSetManifestEmitter`
- Place in the `Backend.Ddl` project since it depends on DDL-specific knowledge (statement counting rules)
- Use `System.Security.Cryptography.SHA256` for hashing
- The emitter is a static class with an `Emit(...)` method returning a string

**Contract type (record):**

```csharp
// As implemented — the emitter computes hash and statement count internally from raw SQL text.
public sealed record DdlManifestEntry(SqlDialect Dialect, string SqlText);
```

### Phase 3: Integrate into `DdlEmitCommand`

**File:** `src/dms/clis/EdFi.DataManagementService.SchemaTools/Commands/DdlEmitCommand.cs`

**Changes:**
- After the per-dialect emission loop, call `DdlManifestEmitter.Emit(...)` with the collected per-dialect SQL strings
- Write `ddl.manifest.json` to the output directory
- Add it to the `emittedFiles` list for the summary output

**Integration point** (in `Execute` method, after the `foreach (var dialect in dialects)` loop):

```csharp
// Emit DDL manifest (DMS-953)
var ddlManifest = DdlManifestEmitter.Emit(effectiveSchemaInfo, ddlEntries);
var ddlManifestPath = Path.Combine(outputDir, "ddl.manifest.json");
WriteFileWithUnixLineEndings(ddlManifestPath, ddlManifest);
emittedFiles.Add("ddl.manifest.json");
```

Where `ddlEntries` is a list of `(SqlDialect, string combinedSql)` collected during the per-dialect loop.

## Risks / Open Questions

1. **Statement counting edge cases:** The simplistic `;\n` counting assumes all statements are terminated by `;`. If any statement uses a different terminator, the count could be off. The emitters are fully controlled, so this should be safe.
2. **MSSQL `GO` handling:** SQL Server DDL uses `GO` batch separators for trigger/view definitions. These should count as statement boundaries but not as statements themselves. Need to clarify whether `GO` itself counts as a statement or just a separator.
3. **Future extensibility:** The manifest spec says "any additional stable metadata needed for troubleshooting (optional)." We start minimal and can add metadata later.
