# Bug: Relational GET-many omits `studentUniqueId` query capability for reference-derived identity alias paths

## Description

Found during [DMS-1159](https://edfi.atlassian.net/browse/DMS-1159) smoke-test conversion (scheduled relational smoke run `27046138701`, NonDestructive API GET-all sweep). The defect is on main and is a compile-time query-capability gap only; the relational write path, hydration, and runtime query planner are unaffected.

## Symptom

Deterministic HTTP 501 NotImplemented on relational GET-many for 15 student-securable DS 5.2 resources. The mappingset manifest records the omission as `unsupported_query_fields` with reason `studentUniqueId: did not resolve to a deterministic relational binding` (30 omission blocks in the DS 5.2 golden: 15 resources x 2 dialects). `StudentAssessmentRegistration` additionally reports `scheduledStudentUniqueId`.

Affected resources:

- `CourseTranscript`
- `Grade`
- `StudentAssessmentEducationOrganizationAssociation`
- `StudentAssessmentRegistration` (`studentUniqueId` and `scheduledStudentUniqueId`)
- `StudentAssessmentRegistrationBatteryPartAssociation`
- `StudentCTEProgramAssociation`
- `StudentHomelessProgramAssociation`
- `StudentLanguageInstructionProgramAssociation`
- `StudentMigrantEducationProgramAssociation`
- `StudentNeglectedOrDelinquentProgramAssociation`
- `StudentProgramAssociation`
- `StudentSchoolFoodServiceProgramAssociation`
- `StudentSection504ProgramAssociation`
- `StudentSpecialEducationProgramAssociation`
- `StudentTitleIPartAProgramAssociation`

The sample extension fixture has analogous alias-pattern fields (e.g. `BusRoute.staffUniqueId` -> `$.staffReference.staffEducationOrganizationAssignmentAssociationUniqueId`).

## Root Cause

MetaEd emits mangled `queryFieldMapping` JSONPaths for person unique-id query fields whose identity arrives through a reference-derived path. The paths do not exist in any document shape, so they match neither root-column `SourceJsonPath` lookups nor the exact reference identity/reference paths that `ReferenceIdentityQueryTargetResolver` uses for ambiguity collapse. `RelationalQueryCapabilityCompiler.CompileQueryField` therefore classifies them `UnmappedPath`, which omits the whole resource's query capability and produces the runtime 501.

The 16 affected query fields take exactly two shapes:

| Shape | Alias path leaf | Example |
|---|---|---|
| **Through-reference** (6 fields): student identity arrives via an intermediate reference | `lowerCamel(TargetResourceName) + "UniqueId"` | `CourseTranscript.studentUniqueId` -> `$.studentReference.studentAcademicRecordUniqueId`; the model has binding `$.studentAcademicRecordReference` -> `StudentAcademicRecord` with `IdentityJsonPath` `$.studentReference.studentUniqueId` and column `StudentAcademicRecord_StudentUniqueId`. Role-named variant: `StudentAssessmentRegistration.scheduledStudentUniqueId` -> `$.scheduledStudentReference.studentEducationOrganizationAssessmentAccommodationUniqueId` |
| **Direct-site, superclass-mangled** (10 fields; the program associations, all subclasses of `GeneralStudentProgramAssociation`): the resource has a direct `studentReference`, but the alias leaf is derived from the resource's superclass | `lowerCamel(SuperclassResourceName) + "UniqueId"` | `StudentCTEProgramAssociation.studentUniqueId` -> `$.studentReference.generalStudentProgramAssociationUniqueId`; the model has direct binding `$.studentReference` -> `Student` with `IdentityJsonPath` `$.studentUniqueId`, `ReferenceJsonPath` `$.studentReference.studentUniqueId`, column `Student_StudentUniqueId` |

The relational model already carries deterministic propagated root-table binding columns for every affected field, and the runtime keyset planner already compiles such predicates correctly (`RelationalQueryPageKeysetPlannerTests` proves `$.studentReference.studentAcademicRecordUniqueId` -> `r."StudentAcademicRecord_StudentUniqueId" = @studentUniqueId`). Only capability compilation refuses the mapping.

## Fix Shape

Bridge the alias paths to existing deterministic root binding columns during capability compilation, with no query-time joins and no change to the root-table-only query contract or runtime planner contracts.

### Alias bridge rule

Add an alias resolution step to `ReferenceIdentityQueryTargetResolver`, consumed by `RelationalQueryCapabilityCompiler` only after the existing ladder misses (after the `ArrayCrossing` and `NonRootTable` classifications, replacing the final fall-through to `UnmappedPath`). Resolution is tri-state:

- `NoMatch` -> continue existing classification flow (`UnmappedPath`).
- `Match` (single candidate group) -> resolve through the existing `ResolveTargetOrThrow` machinery (representative binding column, scalar/descriptor-kind validation) and the existing query-type compatibility check.
- `Ambiguous` (multiple candidate groups) -> mark unsupported with `RelationalQueryFieldFailureKind.AmbiguousRootTarget`.

A candidate matches only when every component below validates against model metadata. The alias query path must have exactly two property segments (`$.{aliasParent}.{aliasLeaf}`) and `aliasLeaf` must end with `UniqueId`.

Role prefix derivation (per binding site): with `refLeaf` = last property of `ReferenceObjectPath` and `baseRefName` = `lowerCamel(TargetResourceName) + "Reference"`:

- `refLeaf == baseRefName` -> role is empty;
- `refLeaf == role + upperCamel(baseRefName)` with non-empty `role` -> that prefix is the role;
- otherwise the site is not alias-bridgeable.

`roleAdjust(name)` = `name` when the role is empty, else `role + upperCamel(name)`.

**Through-reference shape** (candidate `IdentityJsonPath` has exactly two property segments `$.{identityParent}.{identityLeaf}`, `identityLeaf` ends with `UniqueId`):

1. `aliasParent == roleAdjust(identityParent)`
2. `queryFieldName == roleAdjust(identityLeaf)`
3. `aliasLeaf == lowerCamel(TargetResourceName) + "UniqueId"`

**Direct-site shape** (candidate `IdentityJsonPath` has exactly one property segment `$.{identityLeaf}`, `identityLeaf` ends with `UniqueId`):

1. The `ConcreteResourceModel` carries superclass metadata (see below); without it the shape never matches.
2. Alias path parent equals the candidate's `ReferenceObjectPath` (full canonical equality).
3. `queryFieldName == roleAdjust(identityLeaf)`
4. `aliasLeaf == lowerCamel(SuperclassResource.ResourceName) + "UniqueId"`
5. The bridged candidate is the actual local propagated identity column (the candidate's `ReferenceJsonPath` binding, e.g. `$.studentReference.studentUniqueId` -> `Student_StudentUniqueId`).

No broad name-convention matching: every component is validated against binding metadata (`ReferenceObjectPath`, `IdentityJsonPath`, `TargetResource`) or ApiSchema-derived superclass metadata. There is no physical-column-name matching and no hardcoded resource allowlist.

### Superclass metadata plumb

Add optional superclass metadata to `ConcreteResourceModel` (External contract), populated from ApiSchema during relational model construction (`ExtractInputsStep` already reads `superclassResourceName` into `RelationalModelBuilderContext`; ApiSchema also carries `superclassProjectName`, so the contract carries a `QualifiedResourceName? SuperclassResource`). Set it at the `ConcreteResourceModel` construction site in `BaseTraversalAndDescriptorBindingPass` alongside `SecurableElements`/`QueryFieldMappingsByQueryField`. Query capability compilation must not depend on trigger `SuperclassAliasInfo`. The new property is not emitted into the derived-model manifests, so relational-model/DDL/SQL goldens are unchanged.

### Duplicate and cross-site behavior

- Same binding site, same `ReferenceObjectPath`/`TargetResource`/FK and presence column, members converging to one representative storage column: allowed to collapse to the representative binding column, matching exact-path resolution semantics (`It_should_not_use_reference_identity_aliases_to_collapse_duplicate_root_columns` is replaced by a positive same-site alias duplicate collapse test). DMS-1205 already normalizes this shape at model build.
- Different binding sites / multiple matching candidate groups: unsupported (`AmbiguousRootTarget`). Alias bridging must not collapse across sites.
- Near misses (wrong parent, wrong mangled leaf, wrong query field name, missing superclass metadata), array-crossing, non-root-table mappings, and genuinely unmapped fields keep their existing classifications.

## Non-Goals

- No fix to the upstream MetaEd/ApiSchema path generation (tracked separately; the bridge keys off model metadata, not the broken paths' literal text, beyond the two validated mangled-leaf forms).
- No query-time joins, no runtime planner or executor changes, no new failure kinds, no change to descriptor query capability.
- No change to the 501 contract for resources whose fields remain genuinely unsupported.

## Clarifying Questions and Answers

### Questions 1

1. How should the alias-path bridge validate the mangled leaf segment for the 10 program associations (leaf = `generalStudentProgramAssociationUniqueId`, derived from the resource's superclass)? The through-reference shape validates its leaf against the binding's `TargetResource` name, but the direct-site shape has nothing in `ConcreteResourceModel` to validate against today.
2. The guard test `It_should_not_use_reference_identity_aliases_to_collapse_duplicate_root_columns` pins that alias bridging must NOT resolve when one binding site has duplicate identity-binding members (two alias columns converging to one canonical storage column). But the exact-path flow already collapses the identical same-site duplicate shape to the representative column, and DMS-1205 dedups these at model build anyway. Flip this test or preserve the guard?
3. When an alias path matches candidates in more than one reference site (genuinely ambiguous; constructible only synthetically), which failure classification should the unsupported field get?

### Answers 1

1. The direct-site bridge should validate `generalStudentProgramAssociationUniqueId` against schema superclass metadata, not against `TargetResource` and not by a broad `*UniqueId` heuristic. Add optional superclass metadata to `ConcreteResourceModel`, populated from ApiSchema during relational model construction; preserving project/name as a `QualifiedResourceName` is cleaner if `superclassProjectName` is available (it is). Validation rule for the direct-site shape: (1) the `ConcreteResourceModel` has superclass metadata; (2) the query-path leaf equals `lowerCamel(SuperclassResource.ResourceName) + "UniqueId"`; (3) the query-path parent is the same direct reference object path as the matched identity candidate; (4) the candidate being bridged is the actual local propagated identity column; (5) resolution remains unambiguous — multiple candidate groups stay unsupported. Do not depend on trigger `SuperclassAliasInfo`; do not hardcode an allowlist of the 10 resources outside tests. Tests must include one positive direct-site superclass case and negatives for missing superclass metadata, wrong mangled leaf, wrong reference parent, and ambiguity; then regenerate/verify the DS 5.2 golden manifest.
2. Flip that specific test; preserve the safety intent with narrower guards. If the alias bridge has already validated the mangled leaf and proven the query path belongs to the same binding site, duplicate identity-binding members that collapse to one representative column follow the same rule as exact-path resolution. Same binding site / same `ReferenceObjectPath` / same `TargetResource` / same FK and presence column / same representative storage column: allowed to collapse. Different sites, targets, FK columns, or multiple candidate groups: still unsupported/ambiguous. Replace the old guard with a positive same-site alias duplicate collapse test asserting the representative binding column, and keep/add negatives for cross-site collision and ambiguous candidate groups. DMS-1205 strengthens this: the compiler should not guard against a shape the model pipeline no longer treats as semantically distinct, especially when exact-path compilation already accepts it.
3. Use `AmbiguousRootTarget`. The alias bridge found valid schema-shaped matches but more than one deterministic root-table predicate candidate, which maps directly to the existing enum meaning. Implementation is tri-state: `NoMatch` -> existing classification flow (usually `UnmappedPath`); `Match` -> compile supported field; `Ambiguous` -> `AmbiguousRootTarget`. Do not add a new failure kind. Add a synthetic multi-site alias ambiguity test asserting `AmbiguousRootTarget`; existing zero-site cross-site guards remain `UnmappedPath`.
