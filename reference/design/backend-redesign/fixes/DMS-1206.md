# Bug: Relational reference nullability trusts incorrect ApiSchema requiredness, making SurveyResponse un-insertable

## Description

Found during [DMS-1159](https://edfi.atlassian.net/browse/DMS-1159#icft=DMS-1159) populated-template validation (Grand Bend bulk load, Ed-Fi-Data-Standard v5.2.0, BulkLoadClient 7.3.464, EdFiSandbox claim set, DS52+TPDM relational stack). The defect is on main: the [DMS-1159](https://edfi.atlassian.net/browse/DMS-1159#icft=DMS-1159) branch carries no RelationalModel/Ddl changes.

## Symptom

12x HTTP 500 "Relational write failed for resource 'Ed-Fi.SurveyResponse' because the database reported an unrecognized final write failure." The PostgreSQL log shows the underlying error:

```text
null value in column "SurveyResponderChoiceContact_DocumentId" of relation "SurveyResponse" violates not-null constraint
```

Zero SurveyResponse rows landed in the Grand Bend load - the resource is un-insertable on the relational backend.

## Root Cause

`edfi."SurveyResponse"` materializes all three responder choice branches as NOT NULL reference columns (`SurveyResponderChoiceContact_*`, `SurveyResponderChoiceStaff_*`, `SurveyResponderChoiceStudent_*` DocumentId + UniqueId columns). Legacy ODS stores the corresponding SurveyResponse choice columns as nullable columns, with normal nullable FKs and no database-level choice exclusivity constraint.

The DMS relational model currently uses ApiSchema `documentPathsMapping.isRequired` as the source of truth for reference column nullability. For SurveyResponse, MetaEd generates `documentPathsMapping.isRequired = true` for each responder choice member, because the mapping uses the raw requiredness of the referenced source property and does not account for the enclosing MetaEd `Choice`. In contrast, `jsonSchemaForInsert` reflects effective API payload requiredness and does not require any specific responder choice branch.

This creates a DMS-only relational defect: all mutually exclusive choice branches become mandatory database columns even though the API payload schema and legacy ODS relational schema both treat those branch references as optional.

SurveyResponse is the only affected table in the DS52+TPDM surface found by the current information_schema scan for NOT NULL choice-branch columns, but the fix must use schema requiredness semantics, not a SurveyResponse-specific or column-name substring rule.

## Fix Shape

- Change DMS relational model derivation so reference column nullability comes from owning-scope `jsonSchemaForInsert` required-property traversal, matching the existing scalar/descriptor column nullability semantics.
- Keep identity-component references required by validating them against the derived `jsonSchemaForInsert` requiredness. `documentPathsMapping.isRequired` must not be authoritative for DMS relational nullability.
- Leave `Core.ApiSchema.DocumentPath.IsRequired` as raw ApiSchema metadata for existing non-relational consumers, and have the relational extractor populate the existing `DocumentReferenceMapping.IsRequired` from derived `jsonSchemaForInsert` requiredness. Upstream `documentPathsMapping.isRequired` semantics will be fixed shortly but separately.
- Preserve the existing per-reference all-or-none CHECK and composite FK behavior so a selected reference branch must be internally complete and valid.
- Do not add choice-group DB constraints. DMS should continue to mirror legacy ODS relational behavior, which stores choice branches as nullable columns and does not enforce choice exclusivity in database constraints.

## Acceptance Criteria

- DMS relational model compilation derives reference column nullability from owning-scope `jsonSchemaForInsert` requiredness. Optional references are nullable even when `documentPathsMapping.isRequired` is true, while required references and identity-component references remain NOT NULL.
- DS52+TPDM SurveyResponse DDL emits nullable columns for `SurveyResponderChoiceContact_*`, `SurveyResponderChoiceStaff_*`, and `SurveyResponderChoiceStudent_*` reference groups, while retaining their per-reference all-or-none CHECK constraints and composite FKs.
- DS52+TPDM SurveyResponse DDL does not add an at-most-one or exactly-one choice CHECK constraint.
- Relational backend POST succeeds for SurveyResponse payloads with exactly one responder choice branch and for payloads with no responder choice branch, consistent with `jsonSchemaForInsert` and legacy ODS behavior.
- Grand Bend populated bulk load inserts all SurveyResponse rows.

## Clarifying Questions and Answers

### Questions 1

1. Should effective reference requiredness be computed by walking `jsonSchemaForInsert` to the reference object path with the same nullable rules used for scalar/descriptor columns: required at every owning object, no `x-nullable`, and no optional ancestor?
2. Should that derived effective requiredness apply uniformly to the reference `..._DocumentId` column and every propagated identity/descriptor column in the same reference group?
3. If a reference participates in `identityJsonPaths` but derived `jsonSchemaForInsert` requiredness says the reference is optional, should model compilation fail fast instead of forcing the identity-reference group to `NOT NULL`?
4. Should the implementation preserve raw `documentPathsMapping.isRequired` in `DocumentReferenceMapping` for diagnostics or other consumers, or should that field be redefined to mean derived effective requiredness?
5. Should this requiredness change cover references in child collection scopes and extension tables, including optional `_ext` ancestor paths, or is the intended scope limited to core resource references?
6. Should DS52+TPDM SurveyResponse DDL verification update/assert both PostgreSQL and SQL Server authoritative fixtures, or is PostgreSQL-only DDL coverage acceptable for this bug?
7. For the SurveyResponse POST acceptance criteria, should tasking add direct backend integration coverage against generated DS52+TPDM DDL, relational E2E coverage, Grand Bend bulk-load verification, or some required combination of those signals?

### Answers 1

1. Yes. Compute reference effective requiredness from `jsonSchemaForInsert` by walking from the owning table scope to the reference object path using the same rules as scalar/descriptor nullability: every object segment on that path must mark the next property required, no segment may be `x-nullable`, and no optional ancestor within that owning scope may be present. For collection and extension tables, row existence is handled by the table scope; optional array or `_ext` ancestors that only determine whether a row exists should not automatically make columns nullable inside a materialized row.
2. Yes. The derived effective requiredness is group-level requiredness for the whole document-reference group. Apply it to the `..._DocumentId` column and every propagated scalar/descriptor identity binding column for that reference; optional groups remain nullable plus the existing all-or-none CHECK.
3. Yes. If any path in the reference group participates in `identityJsonPaths` but the reference object is not effectively required by `jsonSchemaForInsert`, model compilation must fail fast with a diagnostic naming the resource, reference path, and identity path. Do not force optional identity references to `NOT NULL`.
4. Keep `Core.ApiSchema.DocumentPath.IsRequired` as raw `documentPathsMapping.isRequired` for existing non-relational consumers such as resource load-order calculation, but in the relational model extraction path compute requiredness from `jsonSchemaForInsert` and assign that value to the existing `DocumentReferenceMapping.IsRequired`. Relational column nullability and the identity-reference guard must consume this JSON-schema-derived value.
5. The change applies uniformly to every document reference in the derived relational model: core root tables, child collection/common-type scopes, nested scopes, root extension tables, collection-aligned extension scopes, and extension child collections. Extension row-presence rules remain separate: optional `_ext` controls whether the extension row is materialized, while reference column nullability inside that row follows the owning extension scope's effective JSON Schema requiredness.
6. Update/assert both PostgreSQL and SQL Server DS52+TPDM authoritative outputs. The bug appears through PostgreSQL, but the derived model and DDL contract are cross-engine, so `relational-model.pgsql.manifest.json`, `relational-model.mssql.manifest.json`, `pgsql.sql`, and `mssql.sql` should all reflect nullable SurveyResponse responder-choice reference columns with unchanged all-or-none checks/FKs and no choice CHECK.
7. Tasking should require focused API-level relational integration tests against generated DS52+TPDM DDL for SurveyResponse POSTs with each single responder-choice branch and with no responder-choice branch.
