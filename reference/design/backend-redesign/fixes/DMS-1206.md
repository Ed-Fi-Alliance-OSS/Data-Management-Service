# Bug: Relational reference nullability trusts incorrect ApiSchema requiredness, making SurveyResponse un-insertable

## Description

Found during [DMS-1159](https://edfi.atlassian.net/browse/DMS-1159#icft=DMS-1159) populated-template validation (Grand Bend bulk load, Ed-Fi-Data-Standard v5.2.0, BulkLoadClient 7.3.464, EdFiSandbox claim set, DS52+TPDM relational stack). The defect is on main: the [DMS-1159](https://edfi.atlassian.net/browse/DMS-1159#icft=DMS-1159) branch carries no RelationalModel/Ddl changes.

## Symptom

12x HTTP 500 "Relational write failed for resource 'Ed-Fi.SurveyResponse' because the database reported an unrecognized final write failure." The PostgreSQL log shows the underlying error:

```text
null value in column "SurveyResponderChoiceContact_DocumentId" of relation "SurveyResponse" violates not-null constraint
```

Zero SurveyResponse rows landed in the Grand Bend load - the resource is un-insertable on the relational backend.

## Root Cause

`edfi."SurveyResponse"` materializes all three responder choice branches as NOT NULL reference columns (`SurveyResponderChoiceContact_*`, `SurveyResponderChoiceStaff_*`, `SurveyResponderChoiceStudent_*` DocumentId + UniqueId columns). Legacy ODS stores the corresponding SurveyResponse choice columns as nullable columns, with normal nullable FKs and no database-level choice exclusivity constraint.

The DMS relational model currently uses ApiSchema `documentPathsMapping.isRequired` as the source of truth for reference column nullability. For SurveyResponse, MetaEd generates `documentPathsMapping.isRequired = true` for each responder choice member, because the mapping uses the raw requiredness of the referenced source property and does not account for the enclosing MetaEd `Choice`. In contrast, `jsonSchemaForInsert` reflects effective API payload requiredness and does not require any specific responder choice branch.

This creates a DMS-only relational defect: all mutually exclusive choice branches become mandatory database columns even though the API payload schema and legacy ODS relational schema both treat those branch references as optional.

SurveyResponse is the only affected table in the DS52+TPDM surface found by the current information_schema scan for NOT NULL choice-branch columns, but the fix must use schema requiredness semantics, not a SurveyResponse-specific or column-name substring rule.

## Fix Shape

- Change DMS relational model derivation so reference column nullability comes from owning-scope `jsonSchemaForInsert` required-property traversal, matching the existing scalar/descriptor column nullability semantics.
- Keep identity-component references required by validating them against the derived `jsonSchemaForInsert` requiredness. `documentPathsMapping.isRequired` must not be authoritative for DMS relational nullability.
- Preserve the existing per-reference all-or-none CHECK and composite FK behavior so a selected reference branch must be internally complete and valid.
- Do not add choice-group DB constraints. DMS should continue to mirror legacy ODS relational behavior, which stores choice branches as nullable columns and does not enforce choice exclusivity in database constraints.

## Acceptance Criteria

- DMS relational model compilation derives reference column nullability from owning-scope `jsonSchemaForInsert` requiredness. Optional references are nullable even when `documentPathsMapping.isRequired` is true, while required references and identity-component references remain NOT NULL.
- DS52+TPDM SurveyResponse DDL emits nullable columns for `SurveyResponderChoiceContact_*`, `SurveyResponderChoiceStaff_*`, and `SurveyResponderChoiceStudent_*` reference groups, while retaining their per-reference all-or-none CHECK constraints and composite FKs.
- DS52+TPDM SurveyResponse DDL does not add an at-most-one or exactly-one choice CHECK constraint.
- Relational backend POST succeeds for SurveyResponse payloads with exactly one responder choice branch and for payloads with no responder choice branch, consistent with `jsonSchemaForInsert` and legacy ODS behavior.
- Grand Bend populated bulk load inserts all SurveyResponse rows.
