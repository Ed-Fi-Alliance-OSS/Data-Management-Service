# Story: Convert Remaining Legacy-Backend DMS E2E Tests to Relational Backend

- Issue type: Story.

- Epic: None. DMS-1222 is not assigned to an epic in Jira.

## Description

Convert the remaining DMS E2E source feature scenarios that are not tagged `@relational-backend` so they run against the relational backend. This includes active scenarios and scenarios currently ignored with `@ignore`.

Current source inventory from `src/dms/tests/EdFi.DataManagementService.Tests.E2E/Features`, recomputed from the current branch with Feature/Rule/Scenario tag inheritance:

- 783 DMS E2E scenario definitions total.
- 732 already tagged `@relational-backend`.
- 51 not tagged `@relational-backend`.
- Of those 51, 28 are currently `@ignore` and 23 are not ignored.
- The 2 DMS E2E `Scenario Outline` definitions are already tagged `@relational-backend` and are not part of this conversion set.

This supersedes the earlier 52/29 inventory snapshot. The active scenario count is unchanged at 23. The ignored count includes Rule-level `@ignore` tags, notably the two ignored Rules in `StudentAssessmentAuthorization.feature`.

## Acceptance Criteria

- Every source scenario under `src/dms/tests/EdFi.DataManagementService.Tests.E2E/Features` either has `@relational-backend` coverage or is deliberately removed/replaced with equivalent relational-backend coverage.
- The currently ignored scenarios in the conversion set are reviewed; needed coverage is converted and unignored, while obsolete scenarios are removed or replaced with a documented rationale.
- Converted scenarios preserve the original behavioral intent while using relational-backend setup, data assumptions, and response expectations.
- Converted scenarios are tagged with `@relational-backend` and appropriate relational CI shard tags so `Category=relational-backend` selects them.
- No DMS E2E scenario remains as legacy-backend-only coverage after this story, excluding intentionally non-DMS E2E suites.
- Targeted relational E2E runs use the repository-root `build-dms.ps1 E2ETest` path with the relational environment file, and the run result is recorded on the implementation PR.

## Tasks

1. Convert the 23 active non-relational scenarios to relational-backend execution, updating setup data, payloads, and assertions where backend behavior legitimately differs.
2. Review each active conversion candidate and classify whether it should be converted, replaced, or removed as obsolete.
3. Review all 28 `@ignore` scenarios in the conversion set. For each ignored scenario, document why it is ignored and whether it can be converted to relational-backend coverage.
4. Add `@relational-backend` and relational CI shard tags to converted scenarios.
5. Run targeted relational E2E coverage through `./build-dms.ps1 E2ETest -EnvironmentFile './.env.e2e.relational' -TestFilter 'Category=@relational-backend&...'` or equivalent shard filters.
6. Recompute the feature-file statistics before closing the story and confirm that no source DMS E2E scenarios remain without relational-backend coverage.

## Implementation Plan

Break this story into two separate implementation parts.

### Part 1: Active Non-Ignored Scenarios

Convert the 23 scenarios that are not ignored and not tagged `@relational-backend`.

Default approach:

1. Add `@relational-backend` and the appropriate `@relational-ci-shard-*` tag.
2. Prefer the existing shard convention in the surrounding feature file:
   - Resource create/update/delete/identity scenarios generally use shard 1.
   - Extension and profile read/filtering scenarios generally use shard 2 or 3 based on neighboring scenarios.
   - General URL/query/reference scenarios generally use shard 4.
   - Role/security authorization scenarios generally use shard 3.
3. Run the converted scenarios through the repo-root relational E2E path.
4. Update expectations only when the relational backend legitimately has a different current contract.

Special active-case handling:

- `Resources/DeleteResourcesValidation.feature` includes a documented relational limitation for multi-dependent-resource delete conflict reporting. Scenario 04 currently expects enumeration of both `StudentContactAssociation` and `StudentSchoolAssociation`, while PostgreSQL currently surfaces only the first violated FK. Do not blindly tag this scenario. Replace it with relational-valid coverage, remove it with a DMS-1012 rationale, or move it when DMS-1012 ("Who References Me?" diagnostics) lands.
- `Resources/DeleteResourcesValidation.feature` also contains `Scenario: 00 Background`, a seed guardrail scenario. Review whether it is still needed as a scenario or can be replaced by a true Background or moved to an existing relational delete feature.

### Part 2: Ignored Scenarios

Review the 28 ignored scenarios individually. The goal is not merely to remove `@ignore`; each scenario needs one of these outcomes:

- Converted to relational-backend coverage and unignored.
- Replaced by equivalent relational-backend coverage.
- Removed as obsolete with a documented rationale.
- Left out of this test suite only if the rationale is captured in the PR and linked to an owning backlog item.

Ignored scenario audit buckets:

## Implementation Notes

- Treat this as a test-migration cleanup story, not a backend behavior-change story. The default outcome for a still-valid scenario is conversion to relational coverage.
- Review ignored scenarios individually. Keeping an ignored legacy-only scenario is not sufficient; each one needs converted relational coverage, replacement coverage, or a clear removal rationale.
- Use the repo-root relational E2E path documented in `AGENTS.md` for validation. The `relational-backend` NUnit category selects tests only; it does not configure the DMS container by itself.
- For relational E2E execution, use `.env.e2e.relational` and the `build-dms.ps1 E2ETest` workflow so relational-only provisioning, `RELATIONAL_E2E_DATABASE_NAME`, `dms."EffectiveSchema"`, and DMS restart behavior are applied.
- The build script normalizes filters such as `Category=@relational-backend` to NUnit category names without the `@`. Prefer the repository-root build-script examples in PR validation notes.
- This repo has committed generated `.feature.cs` files. After editing `.feature` tags or scenario text, confirm the compiled NUnit categories are regenerated or otherwise updated before relying on test selection.

## Non-Goals

- Do not convert non-DMS E2E suites as part of this story.
- Do not use direct `setup-local-dms.ps1` plus `dotnet test` as the primary relational E2E signal unless the required relational provisioning helper and test-process database environment variables are also handled.
- Do not preserve legacy-backend-only source scenarios without either relational replacement coverage or documented removal rationale.
