# Story: Implement Security Configuration ProblemDetails

## Description

Implement request-time 500 "Security Configuration Error" ProblemDetails responses for misconfigured authorization metadata per `reference/design/backend-redesign/design-docs/auth.md`.

## Acceptance Criteria
- When a request targets a resource for which no claim set or security metadata has been configured, the system returns a 500 response with:
    - Type: urn:ed-fi:api:system:configuration:security
    - Title: Security Configuration Error
    - Detail: A security configuration problem was detected. The request cannot be authorized.
    - Error: No security metadata has been configured for this resource.
- When the caller's claim matches a resource but no authorization strategies are defined for the requested action, the system returns a 500 response with:
    - Type: urn:ed-fi:api:system:configuration:security
    - Title: Security Configuration Error
    - Detail: A security configuration problem was detected. The request cannot be authorized.
    - Error: No authorization strategies were defined for the requested action '{action}' against resource URIs ['{uri1}', '{uri2}'] matched by the caller's claim
    '{claimName}'.
- When the configured authorization strategy name does not correspond to any known strategy implementation, the system returns a 500 response with:
    - Type: urn:ed-fi:api:system:configuration:security
    - Title: Security Configuration Error
    - Detail: A security configuration problem was detected. The request cannot be authorized.
    - Error: Could not find authorization strategy implementations for the following strategy names: '{strategyName1}', '{strategyName2}'.
- When a custom view-based strategy references a basis entity property that does not exist on the authorization subject entity, the system returns a 500 response with:
    - Type: urn:ed-fi:api:system:configuration:security
    - Title: Security Configuration Error
    - Detail: A security configuration problem was detected. The request cannot be authorized.
    - Error: Unable to find a property on the authorization subject entity type '{targetEntityName}' corresponding to the '{propertyName}' property on the custom authorization view's basis entity type '{basisEntityName}' in order to perform authorization. Should a different authorization strategy be used?
- All scenarios include a correlationId in the response body for traceability, consistent with the ProblemDetails RFC 9457 structure used across DMS.
- These checks run at request time during authorization resolution, not at startup. They are distinct from the startup validations in DMS-1091.
- The error is logged with enough detail for an API host to diagnose and fix the misconfiguration.

NOTE: Some of these checks might have already been implemented.

## Implementation Notes

- Custom view-based authorization strategy execution is not currently implemented in the backend-redesign path. DMS-1099 may standardize request-time security-configuration ProblemDetails for custom view-related failures if an existing shared classifier or staged path already surfaces them, but it must not implement custom view-based authorization, custom view SQL execution, or custom view join/path planning. That work is outside the scope of this story and remains with the view-based authorization stories.
- Change-query runtime route classification is not currently implemented in the backend-redesign path. DMS-1099 must not introduce or change `/deletes` or `/keyChanges` routing. When those endpoints land, they should reuse the DMS-1099 security-configuration ProblemDetails formatter, structured logging, and error-text rules.

## Clarifying Questions and Answers

### Questions 1

1. Should the final wire body for every security-configuration case include the full DMS envelope `status: 500`, `validationErrors: {}`, and `errors: [message]`, with the listed story `Error` text as a single `errors` entry unless multiple deterministic configuration failures are aggregated?
2. For the no-strategies case, should the canonical error text use the story/auth.md wording `against resource URIs ['{uri1}', '{uri2}']`, or preserve the existing implementation wording `against resource ['{resourceClaimName}']` used by current unit tests?
3. Should DMS-1099 cover all request-time authorization surfaces that can return security-configuration failures, including GET-many, GET-by-id, POST, PUT, DELETE, descriptors, and change-query endpoints, or only normal resource CRUD/read surfaces under `src/dms`?
4. Where is the boundary between an ordinary action authorization denial and a security-configuration error: if the caller's claim set has no matching resource/action claim, should that remain a 403 while only matched claims with empty strategy lists become 500?
5. For configured strategy names that are not known built-ins, should DMS first apply the custom view-based `{BasisResource}With...` convention and only use the unknown-strategy security-configuration error when that convention cannot resolve to a known basis resource/descriptor or the resolved strategy is unsupported for the action?
6. For custom view-based path resolution failures, should the listed basis-property error cover all "no join path to the basis resource" cases, or only the specific case where a basis entity primary-key property cannot be found on the authorization subject entity?
7. Should DMS-1099 add only missing mappings and response-shape tests around the existing `FailureResponse.ForSecurityConfiguration` path, or should it also consolidate older generic 500/security-configuration call sites into one shared formatter/logging path?
8. What test level is expected for acceptance: unit tests for response builders and middleware/result mapping only, provider integration tests for relational authorization failures, and/or focused E2E coverage with real claim-set metadata?

### Answers 1

1. Yes. Every DMS-1099 security-configuration response should use the full DMS ProblemDetails envelope from `FailureResponse.ForSecurityConfiguration`: `status: 500`, `validationErrors: {}`, `errors: [...]`, and `correlationId`. Use the listed story/auth.md `Error` text as one `errors` entry unless the request deterministically discovers multiple configuration failures before choosing the response; in that case aggregate those entries in deterministic configured/order-of-discovery order.
2. Use the story/auth.md wording with `against resource URIs ['{uri1}', '{uri2}']`. Populate the URI list from the matched resource claim URI or URIs, in deterministic order. Update existing tests that assert `against resource ['{resourceClaimName}']` to the canonical URI wording.
3. DMS-1099 should cover every currently routed DMS request-time authorization surface that can emit a security-configuration failure through the backend-redesign authorization pipeline: normal resource GET-many, GET-by-id, POST, PUT, DELETE, and descriptor read/write/delete paths. Do not add new strategy semantics or change-query route classification in this story. When change-query `/deletes` and `/keyChanges` runtime routing lands, those surfaces should reuse the same canonical security-configuration ProblemDetails, structured logging, and error-text rules.
4. Keep ordinary permission denial as 403. If the caller's assigned claim set exists but has no matching resource claim, or has no matching action claim for the requested resource/action, return the existing authorization-denied 403. Return the DMS-1099 500 only when security metadata is missing/inconsistent after a claim/resource/action match requires authorization resolution, including an empty strategy list on the matched action. If no claim-set/security metadata can be loaded for the tenant/resource at all, use the `No security metadata has been configured for this resource.` 500.
5. Yes. Classify built-in strategy names first, then apply the custom view-based `{BasisResource}With...` convention before declaring a strategy unknown. A non-built-in name whose basis resource or descriptor resolves for the requested operation is a custom view strategy, not an unknown strategy. Use the unknown-strategy security-configuration error only when the name does not match the convention, the convention's basis cannot be resolved to a known resource/descriptor, or the resolved custom view strategy is not valid for the requested authorization surface.
6. Use the listed basis-property error for custom view strategies whose basis resource/descriptor resolves but no usable authorization join path can be established from the authorization subject to that basis. It is not limited to a direct same-entity property lookup. Choose the missing basis identity/readable property deterministically for `{propertyName}`; if the planner only has a DMS surrogate path failure, use the basis resource's required `DocumentId` anchor in logs and the closest readable basis identity property in the wire error.
7. Use `FailureResponse.ForSecurityConfiguration` as the single wire formatter for DMS-1099 responses. Add missing mappings and response-shape tests around it, and convert existing request-time authorization misconfiguration paths that currently return a generic 500 or inconsistent security-configuration body to this shared formatter/logging path. Do not refactor unrelated non-authorization generic 500 paths in this story.
8. Acceptance should include focused coverage at three levels: unit tests for `FailureResponse.ForSecurityConfiguration`, middleware/action classification, and handler/result mapping; provider integration tests for representative relational security-configuration failures across the currently routed major authorization path families, including resource read/write and descriptor paths; and focused E2E coverage with real CMS claim-set metadata for the externally observable high-risk cases: empty matched strategy list, unknown/custom-view-invalid strategy name, and missing security metadata. Do not require change-query provider integration coverage for DMS-1099 while `/deletes` and `/keyChanges` runtime routing is future work; add that coverage when the change-query runtime and authorization stories land. Do not require an exhaustive E2E or provider-integration matrix across every verb and surface in this story.

### Questions 2

1. Prior authorization slices define security-configuration `errors` entries beyond the four auth.md/DMS-1099 table messages, such as invalid relationship/namespace `AUTH1` payloads, `NamespaceBased` with no usable root Namespace column, SQL Server authorization parameter-limit failures, and relationship strategies with no operation-applicable authorization subjects. Should DMS-1099 preserve those scenario-specific error texts and standardize only the `FailureResponse.ForSecurityConfiguration` envelope, or remap them to one of the four listed messages?
2. For the `No security metadata has been configured for this resource.` case, which request-time claim-set provider outcomes should produce it: an empty returned claim-set catalog, the assigned claim-set name missing from the returned catalog, an assigned claim set with zero resource claims, and/or CMS/cache retrieval failure? Should transport/provider exceptions remain generic system errors instead?
3. When DMS-1099 is implemented, how should known strategy names that are valid metadata but not implemented for a particular authorization surface be handled, especially remaining live-resource/descriptor 501 guardrails and unsupported `ReadChanges` strategies: keep explicit 501 staging responses where sibling stories still own support, or convert them to the unknown-strategy security-configuration 500 on all covered surfaces?
4. Is DMS-1099 expected to update both relational and legacy/non-relational authorization paths, including `Old.Postgresql` and old core validator flows, to the canonical security-configuration ProblemDetails, or only the relational backend and shared core middleware/result-mapping paths used by the backend-redesign flow?

### Answers 2

1. Preserve the scenario-specific `errors` entries already defined by prior authorization slices, and standardize their response envelope through `FailureResponse.ForSecurityConfiguration`. Do not force invalid relationship/namespace `AUTH1` payloads, `NamespaceBased` no-usable-root-column failures, invalid namespace prefixes, SQL Server authorization parameter-cap failures, or relationship no-applicable-subject failures into one of the four auth.md table messages unless the scenario exactly matches that row. The four listed messages remain canonical for those four scenarios; additional request-time authorization misconfiguration diagnostics may remain as stable, deterministic `errors` entries when they identify a specific planner/provider failure.
2. Use `No security metadata has been configured for this resource.` when claim-set metadata lookup succeeds but yields no usable claim-set metadata for the authenticated client: an empty returned claim-set catalog, or an assigned claim-set name that is absent from the returned catalog. Do not use this message for an assigned claim set that exists but has zero resource claims; that follows Answers 1.4 and remains ordinary authorization denial because the caller has no matching resource/action claim. An empty strategy list on a matched action uses the no-strategies security-configuration message. CMS/cache transport failures, provider exceptions, deserialization failures, and timeout-style retrieval failures remain generic system errors with logging, because DMS did not receive a definitive empty metadata result.
3. Keep explicit 501 staging responses for known valid strategy names on live-resource or descriptor authorization surfaces where a sibling story still owns implementation and has intentionally left a fail-closed not-implemented guardrail. DMS-1099 should not reclassify those staged guardrails as unknown-strategy security-configuration failures. For future `ReadChanges` surfaces, follow the change-query stories: strategies unsupported for the `ReadChanges` surface are treated as unavailable strategy implementations and return the security-configuration 500 using `Could not find authorization strategy implementations for the following strategy names: '{strategyName1}', '{strategyName2}'.` DMS-1099 does not implement that mapping before change-query runtime routing exists. As each owning strategy story lands for a surface, remove its temporary 501 and either execute the strategy or reserve security-configuration 500s for true missing, unknown, or invalid metadata.
4. Scope DMS-1099 to the relational backend-redesign authorization path and the shared Core middleware/result-mapping paths used by that flow. Update shared Core response formatting, middleware classification, and backend result mappings even if legacy callers incidentally pass through them. Do not chase `Old.Postgresql`-specific authorization helpers or old validator-only flows unless they already emit a DMS-1099 security-configuration result through the shared path being corrected. Full legacy/non-relational parity should be a separate story if still required after the backend-redesign path is canonical.

### Questions 3

1. In the no-strategies error text, should `{action}` be the exact CMS authorization action name matched for the request, including `ReadChanges` for change-query endpoints, and should `{claimName}` be the matched resource claim `Name`/URI or the API client's assigned claim-set name?
2. For the custom-view basis-property error, what exact values should DMS emit for `{targetEntityName}` and `{basisEntityName}` when the relational backend has resource/table metadata but not ODS CLR entity type names: resource full names, relational table names, or another stable display name?
3. For the acceptance criterion that the error is logged with enough detail to diagnose the misconfiguration, is there a minimum structured logging contract DMS-1099 should implement and test, such as tenant, trace/correlation id, request surface, action, claim set, matched resource claim URI(s), strategy name(s), and provider/planner failure kind, while keeping raw JWTs and provider exception text out of the wire response?

### Answers 3

1. Use the exact CMS authorization action name selected for the request after endpoint/action mapping. Normal resource and descriptor endpoints use `Create`, `Read`, `Update`, or `Delete`. For future change-query route classification, `/deletes` and `/keyChanges` should use `ReadChanges`. `{claimName}` is the matched CMS resource claim `Name` value, i.e., the URI-like resource claim whose action entry has the empty strategy list. It is not the API client's assigned claim-set name. Log the assigned claim-set name separately as diagnostic context.
2. Emit schema-qualified relational resource names formatted as `{schemaName}.{resourceName}` for both placeholders. `{targetEntityName}` is the authorization subject resource being requested, such as `edfi.CourseTranscript`. `{basisEntityName}` is the resolved custom-view basis resource or descriptor selected from the `{BasisResource}With...` convention, such as `edfi.Student` or `edfi.TransportationTypeDescriptor`. For descriptors, use the schema-qualified descriptor resource name in the wire message even when the physical storage table is `dms.Descriptor`; include physical table names only in logs.
3. Yes. Each DMS-1099 response path should emit a structured error log with enough context to diagnose the misconfiguration, following the local logging conventions for event names and scopes. Prefer a stable event name such as `SecurityConfigurationFailure` if it fits the existing logging style, but do not make the exact event name a hard acceptance requirement. The structured context should include the available diagnostic fields that identify the failure: tenant, correlationId, request surface, HTTP method, route/resource path, CMS action, assigned claim set, matched resource claim URI(s), configured strategy name(s), configured strategy index/order when applicable, resource full name, provider/planner failure kind, and the stable wire error message. Custom-view failures should also log target resource full name, basis resource or descriptor full name, missing property/readable property, and any resolved physical table/column path when available. The wire response must remain the canonical ProblemDetails and must not include raw JWTs, secrets, provider exception text, SQL, or stack traces. Tests should assert representative structured diagnostic fields for middleware no-strategies and planner/provider security-configuration failures without requiring every optional field on every path; provider transport exceptions remain generic system errors outside the DMS-1099 contract.
