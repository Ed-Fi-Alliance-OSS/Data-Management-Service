# Story: Implement People-involved Relationship-based Authorization for GET-by-id, POST, PUT, and DELETE

## Description

Implement People-involved relationship-based authorization for GET-by-id, POST, PUT, and DELETE by consuming the shared People relationship authorization core from [DMS-1056](https://edfi.atlassian.net/browse/DMS-1056) and the GET-many People integration from [DMS-1095](https://edfi.atlassian.net/browse/DMS-1095), per:

- `reference/design/backend-redesign/design-docs/auth.md`

## Dependencies

- Depends on [DMS-1095](https://edfi.atlassian.net/browse/DMS-1095).
- DMS-1095 depends on [DMS-1056](https://edfi.atlassian.net/browse/DMS-1056), so this story also inherits the shared People relationship authorization core.

## Acceptance Criteria

- The following relationship-based strategies are implemented for GET-by-id, POST, PUT, and DELETE:
  - `RelationshipsWithEdOrgsAndPeople`
  - `RelationshipsWithEdOrgsAndPeopleInverted`
  - `RelationshipsWithPeopleOnly`
  - `RelationshipsWithStudentsOnly`
  - `RelationshipsWithStudentsOnlyThroughResponsibility`
- GET-by-id: An authorization check is executed using `EXISTS` subqueries against the stored values before reconstitution, and authorization and hydration share the same observed read boundary. If unauthorized, a 403 Forbidden response is returned with appropriate ProblemDetails; no reconstitution occurs.
- POST (new resource): An authorization check is executed against the proposed values from the request body before inserting into `dms.Document`. If unauthorized, the insert does not happen and a 403 Forbidden response is returned.
- POST (existing resource / upsert as update): Authorization follows the same rules as PUT - check stored values first, then check proposed values whenever the People relationship plan has at least one executable proposed subject and the finalized profile-aware rowset can supply that subject's proposed anchor.
- PUT: Two authorization checks are performed:
  - First, authorize using the currently stored values; abort if unauthorized.
  - Second, authorize using the proposed values from the finalized profile-aware rowset whenever the operation has at least one executable proposed People subject, even if those values appear unchanged; abort if unauthorized.
  - Stored authorization, current-state/profile merge work required for proposed authorization, proposed authorization, `If-Match`, guarded no-op, and update DML share a locked write transaction/session or equivalent `ContentVersion` guard/retry boundary.
- DELETE: An authorization check is executed against the stored values before deletion under the same locked write transaction/session or equivalent `ContentVersion` guard/retry boundary used for `If-Match` and delete execution. If unauthorized, the delete does not happen and a 403 Forbidden response is returned.
- People-related securable elements (Student, Contact, Staff) use the DocumentId join-path, auth-view, through-responsibility, hint, and parameterization metadata supplied by DMS-1056.
- For the EducationOrganization portion of mixed EdOrg-and-People strategies, use the ODS-parity DMS concrete root-table subject scope established by DMS-1055; child-table EdOrg predicates are not introduced by this story unless explicitly added by a later design change.
- Inverted strategies correctly swap Source/Target filtering in `EXISTS` subqueries for the EducationOrganization hierarchy table.
- When multiple relationship-based strategies are configured for the same resource, they are combined with OR semantics. The `EXISTS` clauses for each strategy are wrapped in parentheses and combined with OR.
- Multiple securable subjects inside one relationship strategy are combined with AND semantics.
- When authorization fails, the provider raises the established `AUTH1` error code with the versioned compact relationship failure-set payload. The payload preserves the emitted auth check index, plan-relative strategy and subject ordinals, value source, and runtime failure kind so C# can map the full failure set to the shared `RelationshipAuthorizationFailure` DTO and generate ProblemDetails.
- When multiple relationship-based OR strategies are configured and authorization fails, all failed OR strategies and failed subjects are preserved in configured order for ProblemDetails formatting, not just the first failure. See `auth.md` "Authorization Failure Hints" for the hint table and formatting rules.
- ProblemDetails follow the structure defined in `auth.md` "ProblemDetails", including:
  - The `type`, `title`, `detail`, `errors`, and `correlationId` fields per RFC 9457.
  - Securable element paths are translated to user-friendly names (e.g., `$.studentReference.studentUniqueId` -> `StudentUniqueId`).
  - When a single securable element is involved, the error uses the singular form; when multiple are involved, the error uses the plural form and lists all elements.
  - EdOrg claims are shown in the error, up to 5 claims followed by `...`.
- Auth checks use the same observed read/write boundary as the dependent work they gate. One provider command or batch is acceptable but not required for GET-by-id, DELETE, PUT, or POST-as-update. POST create-new still authorizes the finalized proposed values before inserting `dms.Document`, and denial must leave no persisted create artifacts.
- Works for both PostgreSQL and SQL Server, using the `dms.throw_error` function in PostgreSQL and the `CAST('AUTH1 - ...' AS INT)` pattern in SQL Server for aborting batches.
- For SQL Server, when the token's EdOrgId list has fewer than 2,000 entries, use a parameterized `IN` clause; otherwise, use a TVP of type `dms.BigIntTable`.
- This story replaces the temporary not-implemented behavior for People-involved relationship-based GET-by-id, POST, PUT, and DELETE operations.
- Revalidate current relational exclusions, then re-enable `@relational-backend` and the appropriate `@relational-ci-shard-*` tag only on scenarios whose remaining blocker is People relational GET-by-id, POST, PUT, or DELETE authorization. ## Implementation Notes

- DMS-1158 must remove the current single-record People relationship staging block and extend the single-record relationship authorization SQL path to execute People subjects for GET-by-id, POST, PUT, and DELETE. The existing single-record compiler is intentionally EdOrg-only until this story lands; People CRUD execution must support stored root-to-terminal joins, proposed direct person values, proposed self-person existing target anchors, and proposed transitive first-hop anchors that resolve to terminal person `DocumentId` values before checking the selected People auth view.

## Clarifying Questions and Answers

### Questions 1

1. Should DMS-1158 inherit the EdOrg CRUD same-state contracts from DMS-1056 Slices 2 and 4, meaning GET-by-id authorization and hydration must share the same observed read boundary, and DELETE/PUT/POST-as-update must share a locked write transaction/session or equivalent `ContentVersion` guard, even where earlier drafts said "same DB roundtrip"?
2. Should People CRUD operation ordering and response precedence mirror the EdOrg CRUD slices: operation-neutral planning before request-dependent work, POST empty-claims denial before reference resolution and target detection, PUT/GET-by-id/DELETE empty-claims denial only after target existence, stored-value authorization before request-dependent reference/profile/delete work for existing targets, proposed-value authorization before `If-Match`, guarded no-op, or update DML, and authorization `403` before delete constraint/use errors when both would fail?
3. What canonical SQL shape should DMS-1158 use for People stored-value and proposed-value checks, especially transitive person paths: stored root-to-terminal joins with `EXISTS` against the selected auth view, proposed scalar/subqueries from resolved first-hop `DocumentId` values in the finalized rowset, the GET-many root `DocumentId IN` pattern, or another provider-neutral shape?
4. For PUT and POST-as-update proposed-value People checks, what is the exact execution gate: run whenever the finalized profile-aware rowset can produce at least one executable proposed People subject even if values appear unchanged, skip only operation-ineligible subjects such as create-new self-person `DocumentId`, or run only when identifying authorization values changed?
5. Should DMS-1158 reuse the versioned compact `AUTH1` failure-set payload/parser and external `RelationshipAuthorizationFailure` DTO from the EdOrg CRUD slices for People checks, preserving every failed OR strategy/subject and runtime failure kind, rather than emitting only the simple `Unauthorized, index: N` strategy-index message described in earlier drafts?
6. When DMS-1158 lands, how should People relationship CRUD compose with other CRUD authorization strategies: use final `auth.md` AND composition for any already-implemented `NamespaceBased`, `OwnershipBased`, or custom view-based CRUD strategies while returning temporary 501 only for known unimplemented strategies, or fail fast whenever any non-relationship CRUD strategy is present?
7. Does DMS-1158 own any security metadata, ApiSchema, claim-set, or E2E fixture updates needed to express ODS-effective People CRUD strategies such as `NoFurtherAuthorizationRequired` for person creates, EdOrg-only primary-association creates, and the `StudentContactAssociation`/responsibility overrides described by Slice 5, or should the implementation assume those inputs are already correct?
8. Should this story restore every previously listed E2E scenario exactly, including the TokenIntrospection and StudentAssessmentRegistrationBatteryPartAssociation scenarios, or should tasking first revalidate current exclusions and restore only scenarios whose failure is directly caused by People GET-by-id, POST, PUT, or DELETE authorization?

### Answers 1

1. Yes. DMS-1158 should inherit the EdOrg CRUD same-state contracts. GET-by-id People stored-value authorization and hydration must share the same observed read boundary, implemented with a single statement/CTE, provider batch, read transaction/isolation choice, or `ContentVersion` guard/retry. DELETE, PUT, and POST-as-update must use a shared locked write transaction/session or equivalent `ContentVersion` guard/retry so stored authorization, proposed authorization when applicable, precondition checks, guarded no-op, and mutation observe the same target state. The acceptance criteria should use this read/write boundary requirement; one provider command is acceptable but not required.

2. Yes. Mirror the EdOrg CRUD operation ordering and response precedence. Run operation-neutral planning first; true security-configuration failures return the canonical 500 and known unimplemented strategy composition returns the explicit temporary 501 before request-dependent work. For POST, an empty normalized EdOrg claim list returns relationship-not-authorized/403 before reference resolution or create-vs-existing target detection. For GET-by-id, PUT, and DELETE, preserve normal not-found behavior first; for an existing target with empty claims, return 403 before reconstitution, request-dependent reference/profile work, guarded no-op, or mutation. Existing-target PUT and POST-as-update must run stored-value authorization before request-dependent work, then proposed-value authorization before `If-Match`, guarded no-op, or update DML. DELETE must return authorization 403 before `If-Match`, delete execution, or delete constraint/use errors when authorization would fail.

3. Use single-record `EXISTS` checks, not the GET-many root `DocumentId IN` page/count shape. Stored People checks should anchor on the target root `DocumentId`, traverse the resolved root-to-terminal person `DocumentId` path when needed, and test the terminal `Student_DocumentId`, `Contact_DocumentId`, or `Staff_DocumentId` against the selected auth view with the standard token EdOrg predicate. Proposed People checks should bind from the finalized root row/rowset values that will be persisted: direct person references use the proposed terminal person `DocumentId`; transitive person paths use the proposed first-hop `DocumentId` and join from that intermediate row to the terminal person `DocumentId`; self person create-new subjects are operation-ineligible because the new root `DocumentId` does not exist yet. Missing proposed references that fail normal reference resolution remain reference-resolution errors; null or unavailable finalized authorization values that reach the check produce proposed element-required metadata. EdOrg subjects inside mixed strategies keep the EdOrg direct-claim-match plus hierarchy lookup behavior.

4. Run proposed-value People authorization whenever the update operation's People relationship plan has at least one executable proposed subject and the finalized profile-aware rowset can supply that subject's proposed anchor, even if the value appears unchanged from stored data. Do not use changed-value comparison as the gate in DMS-1158. Omit only operation-ineligible subjects, such as create-new self person `DocumentId`, from that operation's AND check. If a selected relationship strategy has no executable subjects after operation eligibility filtering, treat it as the existing no-applicable-subject security-configuration failure unless the operation's effective security metadata did not select that relationship strategy.

5. Yes. Reuse the EdOrg CRUD versioned compact `AUTH1` failure-set payload/parser and the external `RelationshipAuthorizationFailure` DTO, extended with the People metadata supplied by Slice 5. DMS-1158 should preserve the full failed relationship OR group in configured order, including stored/proposed value source, plan-relative strategy and subject ordinals, person kind, auth view/output column, hint metadata, and runtime failure kind such as no claims, no relationship, stored-value invalid data, or proposed element required. The simple `Unauthorized, index: N` strategy-index message must not be reintroduced as the failure contract; invalid or unmappable payloads remain canonical security-configuration 500 failures.

6. Use capability-based composition with final `auth.md` semantics. People relationship strategies join the relationship OR group, and any already-implemented CRUD strategies such as `NamespaceBased`, `OwnershipBased`, or custom view-based authorization compose as AND checks with that OR group in the final execution order. Known CRUD strategies that have not landed yet must return the explicit temporary 501 fail-closed result during planning; DMS-1158 must not silently ignore them or partially authorize with only the People relationship subset. `NoFurtherAuthorizationRequired` remains a no-op, and true security-configuration failures still win over temporary staging results.

7. DMS-1158 owns the audit and any minimal updates needed for the runtime security metadata, ApiSchema-derived authorization inputs, claim-set fixtures, and E2E setup data required to express the ODS-effective People CRUD strategies. It should not hardcode same-row People bootstrap behavior in runtime authorization. The authoritative inputs must express the ODS parity rules from Slice 5: person create operations use `NoFurtherAuthorizationRequired` where that is the effective metadata, primary relationship creates use EdOrg-only authorization in that domain, `StudentContactAssociation` uses its `RelationshipsWithStudentsOnly` override, and `StudentEducationOrganizationResponsibilityAssociation` follows its `relationshipBasedData` metadata. If those inputs are already correct, DMS-1158 only consumes them; if they are missing or stale in the relational test/runtime artifacts, DMS-1158 should update them enough for the restored scenarios to exercise the intended strategies.

8. Tasking should first revalidate the current relational exclusions and restore only scenarios whose missing relational tags are directly caused by People GET-by-id, POST, PUT, or DELETE authorization now implemented by DMS-1158. Do not blindly restore unrelated scenarios just because they appeared in earlier acceptance lists. The People authorization feature scenarios listed for GET-by-id, POST, PUT, DELETE, Contact, Staff, Student, through-responsibility, and role-named People resources should be restored when their setup and assertions are People CRUD-dependent. `Features/Security/TokenIntrospection.feature` scenarios 02 and 03 should not be DMS-1158 restore targets unless revalidation proves their relational exclusion is caused by People CRUD setup. `Features/References/StudentAssessmentRegistrationBatteryPartAssociation.feature` scenario 01 should be restored only if its remaining blocker is People CRUD authorization rather than namespace, reference, profile, or unrelated relational behavior. Later tasking should trim or move any non-People-CRUD retags to the owning story.

### Questions 2

1. For stored and proposed transitive People checks, should the DMS-1158 same-state/read-write contract extend beyond the target document to intermediate referenced rows and selected auth-view membership, or is it sufficient to evaluate the root-to-terminal join and auth view in the authorization statement without locking, revalidating, or version-guarding those intermediate relationship rows through hydration, no-op success, mutation, or create commit?
2. What minimum DMS-1158 test matrix is required across planner/unit, SQL generation, PostgreSQL and SQL Server backend integration, and E2E coverage so tasking avoids a combinatorial explosion while still proving Student, Contact, Staff, through-responsibility, direct/self/transitive paths, no-claims, missing auth view, stored invalid data, proposed element required, and mixed EdOrg-and-People AND/OR behavior?
3. If the relationship ProblemDetails hardening slice has already landed, should DMS-1158 only feed People CRUD failures into the existing parser/formatter and add People endpoint-level assertions, or does DMS-1158 own any missing People-specific ProblemDetails parser, formatter, hint, and wire-response work that was unit-only or out of scope before People CRUD endpoint execution?

### Answers 2

1. Do not extend the DMS-1158 same-state contract to lock, version-guard, or revalidate every intermediate referenced row or selected People auth-view membership through hydration, no-op success, mutation, or create commit. The target document still follows Answer 1.1: GET-by-id must authorize and hydrate under the same observed target read boundary, and DELETE/PUT/POST-as-update must guard the existing target state through the write flow. Intermediate root-to-terminal rows and auth views should be read by the authorization statement itself under normal provider statement/transaction semantics. If that statement observes a null or broken stored path, map it to stored-value invalid data; if a finalized proposed anchor cannot produce the needed person `DocumentId`, map it to proposed element required; if the terminal person is not related through the selected auth view, map it to no relationship. Do not add a post-authorization recheck of auth-view membership before returning, no-op success, update/delete execution, or create commit. For proposed update checks, any reference/descriptor resolution output that feeds the proposed People authorization anchor must either run inside the guarded write boundary or be revalidated there before authorization and mutation, matching the EdOrg Slice 4 contract.

2. Use a layered minimum matrix and do not multiply every person kind by every operation at every layer. Planner/unit coverage should prove all five People relationship strategies, selected subject kinds, auth view and output-column selection, through-responsibility selection, inverted EdOrg metadata, zero-hop self subjects, direct person references, transitive person paths, child-collection People paths skipped as ineligible, create-new self-person ineligibility, no-claims synthetic failure DTOs, missing selected People auth views, no-applicable-subject configuration failures, compact `AUTH1` ordinal mapping, stored/proposed value-source metadata, and mixed EdOrg-and-People AND plus relationship-strategy OR composition. SQL-generation coverage should prove PostgreSQL and SQL Server single-record `EXISTS` shapes for stored and proposed checks across one self Student subject, one direct person reference, one transitive Student path, one Contact path, one Staff path, one responsibility Student path, mixed EdOrg-and-People AND, multiple OR strategies, null stored/proposed subject failure-kind branches, and SQL Server scalar-vs-TVP EdOrg parameterization. Backend integration coverage should run on both PostgreSQL and SQL Server and cover all four CRUD operation classes at least once: authorized and unauthorized GET-by-id with no reconstitution on denial; authorized and unauthorized POST-create with no persisted create artifacts on denial; PUT or POST-as-update stored-value denial, proposed-value denial, and unauthorized no-op denial with unchanged rows; and authorized/unauthorized DELETE with no delete on denial. Across those provider tests, include Student, Contact, Staff, through-responsibility, one direct/self path, one transitive path, empty EdOrg claims, a synthetic missing-auth-view security-configuration case, a SQL-seeded stored invalid-data case, a proposed element-required case, and mixed EdOrg-and-People cases where EdOrg passes/People fails, People passes/EdOrg fails, and a second OR relationship strategy authorizes. E2E coverage should stay focused on real claim-set/token wiring and restored scenario confidence: retag only revalidated People CRUD-dependent scenarios, and ensure at least one authorized/unauthorized stored-value read or delete, one authorized/unauthorized proposed-value create, one update path, one Contact path, one Staff path, and one through-responsibility path are exercised. Leave the broad combinatorics to planner, SQL-generation, and backend integration tests.

3. If the relationship ProblemDetails hardening slice has landed and its shared parser/formatter already supports the People metadata carried by Slice 5, DMS-1158 should feed People CRUD failures into that shared path and add People endpoint-level assertions rather than creating a People-specific formatter. DMS-1158 still owns closing any People-specific gaps that become endpoint-visible when CRUD execution is wired: adapting People failure payloads into the external `RelationshipAuthorizationFailure` DTO, preserving person kind, auth view/table, output column, readable securable names, hints, value source, no-claims context, stored invalid-data, proposed element-required, and no-relationship failure kinds, and asserting the final wire ProblemDetails for representative People stored and proposed failures. If the landed hardening work only had unit-level People DTO coverage or left People endpoint execution out of scope, DMS-1158 must extend the shared parser/formatter/tests enough for People CRUD responses to match `auth.md`; it should not defer those missing wire-response gaps to another already-completed slice.
