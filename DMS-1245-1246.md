# DMS-1245 / DMS-1246 Scrum Planning Cheat Sheet

## Thirty-second overview

DMS-1246 defines the optional `dms.DocumentCache` projection that materializes a
caller-agnostic JSON representation from the relational store. DMS-1245 defines the
CDC/Kafka capability that captures those projected document changes from PostgreSQL or
SQL Server and publishes a stable public stream through Kafka Connect.

DMS-1246 establishes the representation and correctness boundary; DMS-1245 consumes DMS-1246's data,
health, and target observations. Some CDC work can proceed in parallel, especially the
Kafka Connect transform and template code, but supported end-to-end streaming depends on
the DocumentCache projection being operational.


## Suggested implementation conversation

1. Start the parallel foundation work: DMS-1310, DMS-1311, and DMS-1322.
2. Build the projection core through DMS-1312, DMS-1313, and DMS-1314 while adding the
   provider CDC objects in DMS-1320.
3. Use the completed projection and provider pieces to deliver DMS-1315, DMS-1316,
   DMS-1321, and DMS-1319.
4. Add operational enablement and verification through DMS-1323, DMS-1324, and DMS-1318.
5. Finish with cross-feature validation and operator guidance in DMS-1317, DMS-1325, and
   DMS-1326.

## DMS-1246 — DocumentCache projection

### DMS-1308 — `dms.DocumentCache` Projection (Epic)

This epic delivers the complete optional document-projection capability across both
database providers. It covers schema, configuration, materialization, safe persistence,
reconciliation, reads, health, integration testing, runbooks, and representation
restamping; it is also a prerequisite for supported DMS-1245 streaming.

### DMS-1310 — Finalize DocumentCache Schema and Provider DDL

This story creates the PostgreSQL and SQL Server schema foundation for DocumentCache and
integrates it into create-only provisioning, manifests, introspection, and DB-apply
fixtures. It is the first DMS-1246 technical dependency and unblocks the materializer, writer,
projector, read path, and provider CDC setup.

### DMS-1311 — Add DocumentCache Configuration and Target Selection

This story adds strongly typed configuration plus target normalization, validation,
refresh, and lifecycle management for each data store. It can proceed alongside DMS-1310
and later supplies target contexts and observations to projection workers, health, and
CDC readiness.

### DMS-1312 — Add Reusable Caller-Agnostic Document Materialization

This story builds one reusable service that reconstructs the pre-profile document,
composes its served ETag, and verifies source coherence. Reconciliation, optional direct
fill, cache persistence, and representative CDC fixtures all consume this materializer.

### DMS-1313 — Implement Monotonic Cache Upsert and Post-Delete Fencing

This story implements the shared provider-specific cache writer with ordering checks that
prevent stale work from overwriting newer state or resurrecting a deleted document. Its
main planning risk is cross-provider concurrency correctness, so PostgreSQL and SQL
Server interleaving tests are central acceptance evidence.

### DMS-1314 — Add the Asynchronous DocumentCache Reconciliation Loop

This story adds the hosted per-data-store workers that discover differences, materialize
documents, persist corrections, run full audits, and expose an administrative recovery
entry point. It brings together DMS-1310 through DMS-1313 and becomes the source of
projection state for reads, health, and CDC readiness.

### DMS-1315 — Add Fresh-Cache Reads with Relational Fallback

This story lets GET and query processing use a cache entry only when it is known to be
fresh, while keeping relational reconstitution as the correctness fallback. It must
preserve authorization and response shaping in both paths and may optionally direct-fill
the cache through the shared materializer and writer.

### DMS-1316 — Add DocumentCache Health, Readiness, and Telemetry

This story exposes per-data-store projection status using target, prerequisite, audit,
scheduling, and safety observations. It also adds sanitized logs and metrics and provides
the DMS-owned projection status that DMS-1245 combines with connector and Kafka observations.

### DMS-1317 — Add DocumentCache Integration Coverage and Runbooks

This story proves the completed DMS-1246 capability across PostgreSQL and SQL Server with
restart, failure, repair, rebuild, and mixed-target scenarios. It also publishes verified
operator guidance and is the final cross-feature confidence gate for DocumentCache.

### DMS-1318 — Add an Out-of-Band Representation Restamp Utility

This story provides a resumable administrative command for cases where the serialized
representation changes without a normal resource write. It previews and records work,
updates canonical content stamps and Change Query mirrors safely, then verifies that
projection and downstream CDC can reconcile the corrected representation.

## DMS-1245 — Relational CDC/Kafka streaming

### DMS-1309 — Relational CDC/Kafka Streaming (Epic)

This epic delivers opt-in relational CDC for PostgreSQL and SQL Server through provider
objects, Kafka Connect configuration, a DMS-specific transform, bootstrap controls,
contract tests, E2E coverage, and runbooks. It depends on the DMS-1246 projection for the
public document state and on the bootstrap work for local connector registration.

### DMS-1319 — Add Deployment-Owned CDC Binding and Readiness

This story adds durable deployment state for binding a data store to its provider source,
connector, Kafka topics, and source history. It combines DMS projection observations with
provider, connector, and Kafka observations into per-target readiness without changing
normal DMS request routing.

### DMS-1320 — Emit/Provision Provider CDC Key and Database Support

This story provisions the PostgreSQL and SQL Server capture, key, heartbeat, security,
and metadata objects needed by relational CDC. It depends on the ordinary
DocumentCache/source schema from DMS-1310 and gives readiness checks the provider
position and continuity information they need.

### DMS-1321 — Generate PostgreSQL and SQL Server Connector Templates

This story renders validated provider-specific Kafka Connect configurations using the
deployment binding, provider objects, topic policy, transform, heartbeat, metrics, and
offset settings. Template work can begin early, but runnable qualification depends on
DMS-1320 and the transform artifact from DMS-1322.

### DMS-1322 — Add the Relational `DocumentState` Kafka Connect Transform

This story implements and packages the DMS-specific transform that turns provider CDC
records into public document upserts, tombstones, and progress records. It is intentionally
separate from the existing generic expand-JSON transform and must normalize all retained
heartbeat shapes to the fixed non-null progress key.

### DMS-1323 — Add Explicit Local/Bootstrap Connector Registration

This story supplies the command and controller workflow that provisions prerequisites,
topics, ACLs, and the shared Connect offset store, then registers and validates a
connector. It also owns guarded adoption, restart, source replacement, teardown, and the
same lifecycle integration for the E2E harness.

### DMS-1324 — Add Kafka Message and Source-Routing Contract Tests

This story proves the public v1 stream with fast serialized-record tests, provider
fixtures, real broker scenarios, and a reference consumer continuity fixture. The hard
inputs are DMS-1320 and DMS-1322; full broker-readiness scenarios also consume the binding
and bootstrap work, while DMS-1312 supplies representative documents as a soft input.

### DMS-1325 — Replace Legacy Kafka E2E Expectations

This story replaces the quarantined legacy KafkaMessaging tests with API-driven
relational CDC scenarios on both PostgreSQL and SQL Server. It is a late integration
story because it needs real capture, connectors, topics, projection behavior, restart
handling, and source-history continuity from the completed DMS-1246/DMS-1245 path.

### DMS-1326 — Add CDC Setup, Monitoring, Recovery, and Security Runbooks

This story publishes verified operator procedures for setup, observation,
troubleshooting, restart, recovery, containment, source replacement, retirement, and
security on both providers. The documentation is checked against the shipped commands,
templates, status output, and tests, so it should land after the implementation surfaces
stabilize even though its outline can be drafted earlier.

## Useful planning distinctions

- DocumentCache is optional, but it is the supported materialized document source for the
  CDC/Kafka capability.
- Relational reads remain the correctness path; cache-backed reads must fall back rather
  than serve an entry whose freshness is uncertain.
- DMS owns document projection health. Deployment tooling owns connector, Kafka, source
  binding, and combined CDC readiness.
- PostgreSQL and SQL Server parity is acceptance scope throughout, not a later follow-up.
- Fast transform, rendering, and contract-test work can be parallelized before the full
  broker-backed and API-driven paths are available.
